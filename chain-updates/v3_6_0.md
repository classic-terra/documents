# Terra Classic - Update v3.6.0 (v13)

**Release Date**: October 2nd, 2025 </br>
**Chain Update Date**: October 20th, 2025

## Key Changes

The PR can be found here: https://github.com/classic-terra/core/pull/558

Here is a summary of the changes:

- **Wasmd Package Unfork - Phase 2**  
  The completion of the wasmd package unfork migration, transitioning Terra Classic from the forked wasmd v0.46.0-classic.1 to the mainline CosmWasm wasmd v0.46.0. This represents the final phase of removing forked dependencies and aligning with upstream CosmWasm standards.

  Key migration components:
  - Migration from forked Wasmd [v0.46.0-classic.1](https://github.com/classic-terra/wasmd/tree/v0.46.0-classic.1) to mainline Wasmd [v0.46.0](https://github.com/CosmWasm/wasmd/tree/v0.46.0)
  - Comprehensive WASM store key remapping to align with upstream key structures
  - Legacy contract store key migration with composite key rebuilding
  - Migration of contract history, secondary index, and sequence keys

- **Enhanced Testing Infrastructure**  
  Addition of extensive test coverage for the WASM migration:
  - Comprehensive migration test suite for all key types
  - DEX operation validation tests
  - Contract state integrity verification
  - Full upgrade flow testing with legacy and mainline wasmd comparison
  - Enhanced upgrade testing scripts with optimized timing and validation

## Implications / Required Actions
- **No Interface Changes**  
  Despite the significant internal migration, this upgrade introduces no changes to the chain's external interface or functionality. All existing applications, smart contracts, and integrations should continue to operate as before without modification.

- **Enhanced CosmWasm Compatibility**  
  This upgrade brings Terra Classic into full alignment with the upstream CosmWasm ecosystem, enabling:
  - Better compatibility with standard CosmWasm tooling
  - Easier integration of future CosmWasm features and improvements
  - Reduced maintenance overhead by using standard upstream components

- **gRPC Contract Query Changes**  
  The QueryContractInfo response structure has been updated to align with upstream CosmWasm standards:
  - A new `address` field has been introduced
  - The `contract_info` field has been shifted from field 1 to field 2

  **New Format (Post-Upgrade):***
  ```json
  {
    "address": "...",
    "contract_info": {
      "code_id": "...",
      "creator": "...",
      "admin": "...",
      "label": "...",
      "created": {
        "block_height": "...",
        "tx_index": "..."
      },
      "ibc_port_id": "",
      "extension": null
    }
  }
  ```
  Includes top-level `address` field and explicit `extension` field. This format represents a full contract query result.

  **Legacy Format (Pre-Upgrade):**
  ```json
  {
    "contract_info": {
      "code_id": "...",
      "creator": "...",
      "admin": "...",
      "label": "...",
      "created": {
        "block_height": "...",
        "tx_index": "..."
      },
      "ibc_port_id": ""
    }
  }
  ```
  Legacy format without top-level `address` field and without explicit `extension` field.
    
  **Impact on Applications:**
  - Applications and dApps relying on REST (LCD) endpoints through libraries such as terra.js, feather.js, and similar clients are **not affected**
  - Applications and tools using gRPC directly, or libraries built on top of gRPC such as CosmJS or cosmES, **must update their code** to call the new QueryContractInfoService and handle the revised response format
