# Post-Mortem: Terra Classic v3.6.1 Security and Feature Update

**Document Date**: 7 December 2025  
**Release Version**: v3.6.1  
**Chain Upgrade Name**: v13_1

---

## Executive Summary

The v3.6.1 release addresses a critical security vulnerability in CometBFT (ASA-2025-003) that could lead to network halts, along with several improvements to legacy contract support and development tooling.

---

## Security Vulnerability: GHSA-hrhf-2vcr-ghch

### Description

The primary security concern addressed in this release is **ASA-2025-003: Invalid BitArray handling can lead to network halt**.

A bug was discovered in CometBFT's handling of `BitArray` objects where there is a mismatch between the `BitArray`'s expected number of `Elems` for the specified number of `Bits`. The vulnerability existed because insufficient validation was applied when processing messages containing these data structures.

### Technical Details

`BitArray` objects are present in several peer-to-peer messages exchanged between nodes. When a malformed `BitArray` with inconsistent `Bits` and `Elems` values was received, nodes would:

1. **Gossip the invalid message to peers** before processing it themselves
2. **Crash or enter an invalid state** when attempting to process the malformed data

This sequence of events could trigger a cascade failure across the network, as nodes would propagate the malicious message before crashing.

### Impact Assessment

| Factor | Rating |
|--------|--------|
| **Criticality** | High |
| **Affected Components** | Validators, Full nodes, Users |
| **Potential Impact** | Network halt (DoS) |
| **Exploitability** | A malicious peer could send specially crafted messages |

In the worst case, a single malicious actor sending these payloads could cause a complete network halt, as nodes would gossip the malicious messages before processing them.

### Resolution

The fix was implemented by upgrading CometBFT from **v0.37.15** to **v0.37.16** upstream version, which includes enhanced validation for `BitArray` objects.

## Oracle Gas Limit DoS Vulnerability

### Background

A critical vulnerability existed in the oracle transaction handling that could be exploited for denial-of-service attacks and potential oracle price manipulation.

**Key characteristics of oracle transactions on Terra Classic:**

- Oracle transactions are **fee-free** and cost no gas to the submitter
- Only bonded validators can submit oracle votes
- Oracle transactions are **prioritized** in block ordering and always processed before user transactions
- The active validator set is currently not filled, making it nearly free to become a bonded validator

### Attack Vector

A malicious actor could exploit this system through the following attack:

1. **Become a validator**: Due to the unfilled active validator set, an attacker could become a bonded validator with minimal cost

2. **Craft malicious oracle messages**: Set the gas limit of oracle vote transactions equal to the block gas limit

3. **Block exclusion attack**: Since a transaction with `gas_limit = block_gas_limit` would consume the entire block's gas capacity, only that single oracle message would be included in the block, effectively excluding all other transactions

4. **Coordinated attack amplification**: If multiple malicious validators (e.g., 5) coordinated this attack, they could:
   - **Block all legitimate validator oracle votes** from being included
   - **Block all user transactions** from being processed
   - **Halt effective chain operation** while the chain technically continues producing blocks

5. **Oracle price manipulation**: By preventing legitimate validators from submitting their price votes, attackers could potentially manipulate the oracle price calculation, as only their crafted votes would be included in blocks

### Severity

| Factor | Rating |
|--------|--------|
| **Criticality** | High |
| **Cost to Attack** | Nearly zero (free oracle txs + cheap validator bonding) |
| **Potential Impact** | Chain DoS, Oracle price manipulation |
| **Affected Parties** | All users, validators, DeFi protocols relying on oracle prices |

### Mitigation

The v3.6.1 release includes fixes to prevent oracle transactions from using gas limits that would monopolize block space, ensuring fair transaction inclusion and protecting oracle price integrity.

## Legacy Contract Fixes

### Query Plugin Improvements

The release includes significant improvements to the WASM query plugin to support legacy Terra query variants:

**File**: `wasmbinding/query_plugin.go`

A size limit check was added to prevent DoS attacks via oversized query requests:

```go
// limit size to 64kb to check for legacy (DoS)
if len(request) > 64<<10 {
    return request
}
```

The plugin now properly handles legacy routed queries, translating them to modern `TerraQuery` format for backwards compatibility with contracts using older query patterns.
